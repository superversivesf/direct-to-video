# v2.0 Game Redesign

**Date:** 2026-07-18
**Status:** Design
**Motivation:** Playtesting revealed the executive-centric flow is too much responsibility for one player and the 1-point-per-round scoring doesn't reward consistently good pitchers. Redesign voting, scoring, round structure, and note cards for a more democratic, replayable game.

## Problem Summary

The current game has these shortcomings:
1. **Executive is a bottleneck** — one player manages timer, notes, AND picks the winner. Too much responsibility, and if they disconnect the game soft-locks.
2. **1 point per round** — doesn't reward players who consistently get votes but don't win the round outright.
3. **Round count tied to player count** — can't play a quick 3-round game with 6 players.
4. **No ready indicators** — note-giver/exec can't tell who's still choosing cards.
5. **No kick** — host can't remove disruptive or AFK players.
6. **Stale disconnects** — players who close the browser linger as "disconnected" forever, blocking game flow.

## Design

### 1. Executive Role Removed → Note-Giver Role

**Remove:**
- `Room.executiveId` — gone
- `Player.isExecutive` — gone
- `Room.executiveNotes` — renamed to `Room.noteGiverNotes`
- All executive-specific logic (exec picks winner, exec 2x vote weight, exec starts voting)

**Add:**
- `Room.noteGiverId: string | null` — the player who gives notes this round
- `Player.isNoteGiver: boolean` — flag for UI
- `Room.noteGiverNotes: Card[]` — the 3 note cards (renamed from `executiveNotes`)
- `Room.noteGiverOrder: string[]` — shuffled rotation queue of player IDs
- `Room.noteGiverIndex: number` — current position in the rotation

**Note-giver selection:**
- On `startGame`, build `noteGiverOrder` by shuffling all player IDs
- Each `setupRound`, pick `noteGiverOrder[noteGiverIndex]` as the note-giver
- Increment `noteGiverIndex`; when it exceeds the list, reshuffle and reset
- Skip disconnected players (advance to next in rotation)
- The note-giver draws 3 note cards in `setupRound`
- The note-giver manages the timer (start/pause/resume) and plays notes during pitches
- The note-giver **pitches last** — `startPitching` sorts them to the end of pitch order (in addition to franchise card holders)

**Socket events affected:**
- `start_timer`, `pause_timer`, `play_note` — check `noteGiverId` instead of `executiveId`
- `select_winner`, `start_voting`, `end_voting` — **removed entirely**
- `cast_vote` — stays, but no exec 2x weight
- New: `set_total_rounds` — host sets round count in lobby

### 2. Fixed Round Count

- Host selects from a dropdown in the lobby: **3, 5, 7, or 10 rounds** (default 5)
- `Room.totalRounds: number` — new top-level field (replaces `round.total`)
- `Room.round.current` stays — increments each round
- `set_total_rounds` socket event — host only, lobby phase only
- `startGame` uses `room.totalRounds` to set `round.total`
- Game ends when `round.current > totalRounds`
- Lobby UI: dropdown next to franchise toggle, host-only

### 3. Voting & Scoring Redesign

**Voting flow:**
1. Last pitcher ends → phase transitions to `round-end`, `votingActive: true` automatically
2. 15-second voting timer starts immediately
3. All players AND audience members see all revealed movies with vote buttons
4. **Players cannot vote for themselves** — their own movie is hidden from their ballot (UI excludes it; server validates `voterId !== playerId` for players)
5. Each vote = 1 point. No weight differences.
6. Timer expires OR all eligible voters have voted → tally votes into scores
7. Each player's `score` increases by their vote count that round
8. Round winner banner shows the player with the most votes this round
9. Round advances to next round (or game-end if last round)

**Scoring:**
- `Player.score` accumulates **raw vote counts** across all rounds (not 1 point per win)
- Example: Round 1, Alice gets 3 votes, Bob gets 2, Charlie gets 1. Scores: Alice=3, Bob=2, Charlie=1. Round 2, Bob gets 4, Alice gets 1, Charlie gets 1. Scores: Alice=4, Bob=6, Charlie=2. Bob wins overall.
- Game-end: highest cumulative score wins. Ties displayed as ties (no tiebreaker).

**Data changes:**
- `Room.votingActive` — set `true` at round-end automatically (not by exec action)
- `Room.votes` — unchanged structure (`Record<voterId, playerId>`)
- `selectWinner` state-machine function → replaced with `tallyVotes` that adds vote counts to each player's score
- `tallyVotes` no longer has exec 2x weight — all votes are 1x
- `castVote` validates: `voterId !== playerId` for players (audience can vote for anyone)
- `endVoting` → renamed to `tallyAndAdvance` — tallies votes into scores, advances round

**Server-side vote tallying (`tallyAndAdvance`):**
```
1. For each vote in room.votes: counts[votedFor] = (counts[votedFor] || 0) + 1
2. For each player: player.score += counts[player.id] || 0
3. Find round winner (highest count this round) for the banner
4. Clear votes, set votingActive = false
5. If round.current >= totalRounds → phase = "game-end"
6. Else → nextRound()
```

**Timer expiry during voting:**
- Same as current: timer tick loop detects expiry, calls `tallyAndAdvance`

**All-voted early termination:**
- After each `castVote`, check if all eligible voters have voted
- "Eligible voters" = all connected players who have a movie this round (they vote for someone else) + all audience members present
- The note-giver is also eligible (they pitch last but still vote)
- If all eligible voters have voted → `tallyAndAdvance` immediately, cancel timer

### 4. Lobby Management

#### Ready Indicators

- During `setup` and `card-selection` phases, the note-giver and audience see writer ready status
- A writer is "ready" if they have a movie in `room.movies` (derived from existing state — no new field)
- UI: green checkmark next to ready writers, "choosing..." for those still selecting
- The note-giver's setup view shows the player list with ready indicators

#### Host Kick Players

- New socket event: `kick_player(playerId: string)` — host only, any phase
- Server handler:
  1. Validate caller is host
  2. Find target player in room
  3. Remove from `room.players`
  4. Remove from `playerSockets` map
  5. Disconnect their socket with an error message: "You have been removed from the room"
  6. If kicked player was note-giver → reassign to next in rotation
  7. If kicked player was host (shouldn't happen, but guard) → promote next
  8. Broadcast updated state
- Kicked player can rejoin with the same name (fully removed, not just disconnected)
- `ClientToServerEvents` gets `kick_player: (playerId: string) => void`

#### Stale Disconnect Handling

- When a player disconnects (socket `disconnect` event), start a **60-second timeout**
- If the player rejoins within 60s → cancel timeout, restore identity (existing behavior)
- If 60s passes without rejoin → **fully remove** the player from `room.players` (not just `isDisconnected: true`)
- If the removed player was note-giver → reassign to next in rotation
- If the removed player was host → promote next connected player
- The timeout lives in a `Map<string, NodeJS.Timeout>` in `sockets.ts` (not serialized)
- On rejoin, the `join_room` handler checks if the player name exists in `room.players` (reconnect) or not (full rejoin as new player)

### 5. Refactoring (folded into Sub-project 1)

The refactor plan from 2026-07-15 should be executed as part of the core gameplay changes:
- Extract `substituteDraws` → `card-ops.ts` (reduces duplication in the new note-giver flow)
- Extract `weightedVoteCounts` → but simplified since all votes are 1x now. The function becomes `voteCounts(room) → Record<string, number>` with no weighting.
- Split `sockets.ts` → `sockets/rate-limits.ts`, `sockets/state-mapper.ts`, `sockets/handlers.ts` (the file will grow with kick/stale logic, so splitting first makes the new code cleaner)
- Extract `promoteNextHost` → `rooms.ts` (already done in the refactor plan)
- Timer state constructors → `shared/timer-helpers.ts` (already done in the refactor plan, reduces duplication in the new voting timer)

## What This Design Does NOT Do

- No team mode (future scope)
- No Writers' Room variant (future scope)
- No tiebreaker mechanism (ties displayed as-is)
- No spectator-only mode beyond audience
- No chat/messaging system
- No persistent leaderboards across games
